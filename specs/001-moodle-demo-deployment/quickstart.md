# Quickstart: Moodle Demo Site Deployment

**Feature**: `001-moodle-demo-deployment`  
**Target time**: ~15 minutes from zero to live HTTPS site  
**Prerequisites**: A GitHub account, a DigitalOcean account, and a registered domain name.

---

## Overview

This guide walks you through the four steps required to deploy the Moodle demo site:

1. Provision a DigitalOcean Droplet
2. Configure DNS
3. Add GitHub repository secrets
4. Push to `main` (triggers automated deployment)

---

## Step 1 — Provision a DigitalOcean Droplet

1. Log in to [DigitalOcean](https://cloud.digitalocean.com) and click **Create → Droplets**.
2. Choose:
   - **Region**: Closest to your audience
   - **OS**: Ubuntu 22.04 LTS x64
   - **Size**: see sizing table below — leave at the recommended default for now; you can resize later if you plan to run multiple Moodle replicas
   - **Authentication**: SSH Key — upload your public key (the one whose private key you'll add to GitHub secrets)

**Droplet sizing guide** (~1 GB RAM + 0.5 vCPU per additional client):
| Clients | Recommended Droplet | vCPU | RAM | Monthly |
|---------|---------------------|------|-----|---------|
| 1–2 | `s-2vcpu-4gb` | 2 | 4 GB | ~$24 |
| 3–5 | `s-4vcpu-8gb` (**recommended starting point**) | 4 | 8 GB | ~$48 |
| 6–10 | `s-8vcpu-16gb` | 8 | 16 GB | ~$96 |
3. Click **Create Droplet** and note the assigned **IP address**.

### Install Docker on the Droplet

SSH into the Droplet and run:

```bash
# Update package index
apt-get update

# Install Docker using the official convenience script (Ubuntu 22.04)
curl -fsSL https://get.docker.com | sh

# Verify
docker --version        # expect: Docker version 24.x or higher
docker compose version  # expect: Docker Compose version v2.x or higher
```

### Configure the Deployment Directory

```bash
mkdir -p /opt/moodle-demo
cd /opt/moodle-demo

# Clone the repository
git clone https://github.com/YOUR_ORG/moodle-demo.git .
```

### Configure the DigitalOcean Cloud Firewall

In DigitalOcean console → **Networking → Firewalls**:
- Allow **inbound TCP port 80** from all sources
- Allow **inbound TCP port 443** from all sources
- Allow **inbound TCP port 22** from your administrator IP only
- All other inbound ports: blocked

---

## Step 2 — Configure DNS

**Each client requires its own DNS A record.** All client domains point to the same Droplet IP.

For each client domain (or your first demo site), add an **A record** in your domain registrar or DNS provider:

| Name | Type | Value | TTL |
|------|------|-------|-----|
| `acme-corp.com` (or your domain) | A | `<Droplet IP>` | 300 |

Wait for DNS propagation (typically 1–5 minutes with TTL 300). Verify:

```bash
dig +short acme-corp.com
# Should return your Droplet's IP address
```

**Important**: Each client domain's DNS must resolve before running `scripts/add-client.sh` for that client. Let's Encrypt's HTTP-01 challenge requires port 80 to be reachable from the internet at the configured domain.

---

## Step 3 — Configure GitHub Repository Secrets

Go to your repository on GitHub: **Settings → Secrets and variables → Actions → New repository secret**.

Add the following **shared infrastructure** secrets (used by `deploy.yml` to write the server’s `.env` file):

| Secret Name | Value to Set |
|-------------|-------------|
| `SSH_HOST` | Your Droplet's IP address |
| `SSH_USERNAME` | `root` (or your deploy user) |
| `SSH_PRIVATE_KEY` | Contents of your SSH private key file (e.g., `~/.ssh/id_ed25519`) |
| `ACME_EMAIL` | Your email for Let’s Encrypt notifications (applies to all client domains) |
| `MARIADB_ROOT_PASSWORD` | Strong random password for MariaDB root |
| `MARIADB_PASSWORD` | Strong random password for the shared `moodle` database user |

> **Per-client credentials** (admin usernames, passwords, domains) are stored in `clients/{name}/.env` on the Droplet and are auto-generated by `scripts/add-client.sh`. They are **not** GitHub secrets.

**Generate strong passwords** (run locally):
```bash
openssl rand -base64 24  # run twice — once for each DB password
openssl rand -base64 24  # Moodle admin password
```

---

## Step 4 — Deploy

Push a commit to the `main` branch:

```bash
git push origin main
```

This triggers the `deploy.yml` GitHub Actions workflow. Watch progress at:

```
https://github.com/YOUR_ORG/moodle-demo/actions
```

The workflow:
1. SSH into the Droplet, generates `.env` from secrets
2. Runs `docker compose pull && docker compose up -d --remove-orphans`
3. Waits for Moodle to pass its health check (first run: ~5–8 minutes for DB install)
4. Runs smoke test (curl HTTPS endpoint)
5. Reports pass ✅ or fail ❌ with logs

**Expected total time from push to live site: ~10–15 minutes** on first deploy.

---

## Step 5 — Verify

Once the workflow reports success:

1. Visit `https://moodle.yourdomain.com` in a browser.
2. Confirm the **Moove theme** is active (polished header, custom colours).
3. Log in with the admin credentials you set.
4. Navigate to **Site administration → Courses** — verify 2 demo courses exist.
5. Log out and **self-register** as a new user to test the visitor journey.

## Updating the Site

To deploy a change (configuration update, image version bump, etc.):

```bash
# Edit files locally, then:
git add .
git commit -m "chore: update configuration"
git push origin main
```

The workflow re-deploys with zero-downtime rolling update. Only changed containers are restarted.

## Step 6 — Provision a New Client

Each client gets an isolated Moodle instance (dedicated container, database, and domain) provisioned by a single script.

### Prerequisites

- Shared infrastructure is running (`traefik`, `mariadb`, `redis`).
- The client’s domain has a DNS A record pointing to the Droplet IP (see Step 2).
- You are SSH’d into the Droplet.

### How to provision

SSH into the Droplet and run:

```bash
ssh root@<DROPLET_IP>
cd /opt/moodle-demo

# Provision a new client (name + domain required)
./scripts/add-client.sh acme-corp acme-corp.com
```

The script will:
1. Create `clients/acme-corp/` with a generated `.env` file (admin credentials, DB name, Redis prefix).
2. Render `clients/_template/docker-compose.yml.tpl` → `clients/acme-corp/docker-compose.yml`.
3. Create a dedicated MariaDB database (`moodle_acme_corp`) for this client.
4. Add an `include:` entry to `docker-compose.yml` for the new client.
5. Start `moodle-acme-corp` and wait for its health check to pass (~5–8 minutes).
6. Run the seed container (`seed-acme-corp`) to apply the Moove theme and create demo content.

**Expected time**: ~10–15 minutes per client (dominated by Moodle’s first-run DB install).

### View generated credentials

```bash
cat clients/acme-corp/.env
# Contains: CLIENT_NAME, CLIENT_DOMAIN, MOODLE_USERNAME, MOODLE_PASSWORD, etc.
```

> Store these credentials securely. The `.env` files are gitignored and never committed.

### Verify the new client is live

```bash
curl -I https://acme-corp.com/login/index.php
# Expect: HTTP 200

docker compose ps moodle-acme-corp
# Expect: running (healthy)
```

### Add additional clients

Repeat the provisioning step for each new client, substituting the correct name and domain:

```bash
./scripts/add-client.sh globex globex.example.com
./scripts/add-client.sh initech initech.example.com
```

Each client is fully isolated from the others. Provisioning one client has no effect on existing clients.

## Troubleshooting

### TLS Certificate Not Issuing

- Verify DNS A record points to the Droplet IP.
- Verify port 80 is open in the DigitalOcean firewall.
- Check Traefik logs: `docker compose logs traefik | grep -i acme`

### Moodle Health Check Timing Out

- First-run installation takes 5–8 minutes. This is normal.
- Check Moodle logs: `docker compose logs moodle`
- If MariaDB is not healthy: `docker compose logs mariadb`

### Workflow Fails at SSH Step

- Confirm `SSH_HOST` secret contains just the IP address (no extra spaces).
- Confirm the public key matching `SSH_PRIVATE_KEY` is in `~/.ssh/authorized_keys` on the Droplet.
- Test manually: `ssh -i ~/.ssh/id_ed25519 root@<IP>`

### Site Accessible but Theme Not Applied

- The seed container may not have completed. Check: `docker compose logs seed`
- Manually apply theme: `docker compose exec moodle php /opt/bitnami/moodle/admin/cli/cfg.php --name=theme --set=moove`
- Then purge caches: `docker compose exec moodle php /opt/bitnami/moodle/admin/cli/purge_caches.php`

### New Client Container Not Starting

- Check container logs: `docker compose logs moodle-acme-corp`
- Confirm MariaDB is healthy and the database was created: `docker compose exec mariadb mysql -u root -p -e "SHOW DATABASES;"`
- Confirm the `clients/acme-corp/.env` file exists and has valid values.
- If the health check times out (first run can take 8+ minutes on smaller Droplets), increase `start_period` in the client’s Compose file.

### Client Site Returns 404 / Traefik Routing Not Working

- Confirm Traefik labels are present in `clients/acme-corp/docker-compose.yml`.
- Confirm the container is running and healthy: `docker compose ps moodle-acme-corp`
- Check Traefik’s routing config: `docker compose logs traefik | grep acme-corp`
- Verify DNS for the client domain resolves to the Droplet IP: `dig +short acme-corp.com`

## Teardown

To stop all services and remove containers (data volumes are preserved):

```bash
docker compose down
```

To also remove all data volumes (DESTRUCTIVE — all Moodle data lost):

```bash
docker compose down -v   # Ask before running this on a production/demo site
```
